# Murmurland（X-Clone 社群平台）

## Proposal Report

### 動機與目標

微型部落格與即時動態已成為資訊傳播與社群互動的主要載體。本專題 Murmurland 旨在以全端 Web 應用實作一套可運作之類 Twitter（X）平台：支援發文、留言、社交關係、主題標籤與即時通知。

### 競品比較


**具體目標：**

- 打造一個完整的現代前後端架構（Next.js App Router + MongoDB）。
- 將「討論串與留言」建模為階層式資料，實作遞迴留言與分層顯示的 UI。
- 將「追蹤關係」建模為有向圖 (Directed Graph)，並用它來產生個人的動態時報。
- 串接 Pusher 達成即時互動，練習事件驅動 (Event-driven) 與非同步的資料處理。

### 使用技術

- **前端：** Next.js（App Router）、React、Tailwind CSS、SWR、TypeScript  
- **後端：** Next.js API Routes（`app/api/`**）  
- **資料層：** MongoDB Atlas、Prisma ORM（schema 與查詢）、MongoDB 原生驅動（部分寫入與原子操作）  
- **即時：** Pusher（伺服器 trigger／客戶端 subscribe）  
- **驗證與媒體：** NextAuth、Cloudinary

### 與課程的關聯

本專題與 DSAP 的連結在於以明確的抽象資料型別與演算法思維處理平台核心問題，說明如下。

**1. 樹狀結構與遞迴（討論串／留言）**  

- 資料庫設計： `Post` 利用 `parentPostId` 和 `rootPostId` 形成森林 (Forest) 或樹狀結構；`Comment` 則靠 `parentCommentId` 形成留言樹。這是標準的以節點與父指標儲存之樹 (parent-pointer tree)，空間上每個節點 `O(1)` 指標，整體空間為 `O(n)`。
- 貼文執行緒 API 會載入根貼文與「第一層」直接回覆；巢狀討論目前主要透過 `/post/[id]/comment/[commentId]` 逐層下鑽（父節點＋子節點一次查詢）。
- 對照課堂中的樹走訪 (Tree Traversal)，如果要在前端一頁展開所有留言，就需要把平面的列表資料「建樹」（例如用 `Map<id, children[]>` 分組後跑 DFS 輸出），時間 `O(n)`、額外空間 `O(n)`；若使用遞迴元件 (Recursive Component) 渲染子樹，深度則會受限於 Stack 的限制。

**2. 圖論（追蹤關係）**  

- 資料庫設計： 使用者之間的 `Follow` 關係 (`followerId → followingId`) 構成了一個有向圖 (Directed Graph)。
- 演算法應用： 在產生「追蹤中」的動態時報時，系統會先查出使用者的「出邊 (Out-edges) 鄰居集合」(`followingIds`)，再去撈這些人的貼文。這相當於對由鄰居所誘導的子圖 (Induced Subgraph) 做查詢。單次查詢複雜度為 `O(|F| + Q)`，其中 `|F|` 為追蹤人數，`Q` 為資料庫掃描與索引的成本。

**3. 佇列與事件驅動（Pusher 即時更新）**  

- 機制： 後端資料一有變動，就呼叫 Pusher 廣播；前端訂閱後觸發畫面更新。這屬於發布/訂閱 (Pub/Sub) 的事件驅動架構。雖然不是自己手刻環形佇列，但訊息在前後端之間形成了非同步事件流，完全呼應課堂上佇列 FIFO (先進先出) 與非同步處理的概念，確保事件依序送達並更新畫面。
- 複雜度：單一頻道綁定註冊為 `O(1)`；前端每次因事件觸發而重新 fetch 資料的成本，可記為 `O(重新抓取之資料量)`。

**4. 排序與搜尋**  

- 時間線合併排序：動態時報需要撈取一般貼文與轉發 (Repost)。如果將兩邊依時間排好序的資料流合併，這就是二路合併 (Two-way Merge) 的思想。目前實作為合併成一個陣列後跑 `sort`，假設元素個數為 `n`，時間複雜度為 `O(n log n)`，空間為 `O(n)`。
